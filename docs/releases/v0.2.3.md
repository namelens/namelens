# NameLens v0.2.3 Release Notes

Release date: 2026-02-23

## Highlights

v0.2.3 is a dogfooding polish release: bug fixes from the v0.2.2 naming
exercise, brand workflow improvements, and provider routing UX. Seven bugs
fixed, three features added.

- **Brand context for review**: `--context-file` and `--scan-dir` pass product
  context through to brand analyses
- **Provider override for generate**: `--provider` enables invocation-scoped
  provider selection for A/B testing across AI backends
- **OSS profile**: Registry-and-handle-only checks for open-source projects

## Brand Context for Review

Brand analyses without product context produce generic assessments. Now you can
pass context directly:

```bash
# From a file (truncated to 2000 chars)
namelens review myproject --mode=brand --depth=deep \
  --context-file ./VISION.md

# Scan a directory for context files
namelens review myproject --mode=brand --depth=deep \
  --scan-dir ./docs

# Control scan budget (default: 32000 chars)
namelens review myproject --mode=brand --scan-dir ./docs --scan-budget 16000
```

The context is injected into `brand-plan` and `brand-proposal` prompts,
producing assessments that reflect your actual product positioning, target
audience, and competitive landscape instead of generic guesses.

### Before and After

Without context, brand-plan for a calendar proxy tool described it as a "CLI
availability checker." With `--context-file` pointing to the product vision,
the same analysis correctly identified it as a "policy-governed calendar proxy
for AI agent workflows" with appropriate positioning in the developer tools
ecosystem.

## Provider Override for Generate

Switch AI providers per-invocation without editing config or using environment
variable overrides:

```bash
# Force Anthropic for this run only
namelens generate "agent gateway" --provider namelens-anthropic

# Force OpenAI with deep depth
namelens generate "agent gateway" --depth deep --provider namelens-openai
```

Provider precedence for `generate`:

1. `--provider` flag (if set)
2. `ailink.routing[<prompt-slug>]` (config routing table)
3. Provider `roles` match
4. `ailink.default_provider`
5. Single enabled provider fallback

Validation gives clear, actionable errors:

- Unknown: `unknown provider "foo" (valid: namelens-anthropic, namelens-openai, namelens-xai)`
- Disabled: `provider "foo" is disabled`

## OSS Profile

Open-source projects rarely need domain checks. The new `oss` profile checks
only package registries and social handles:

```bash
namelens check mylib --profile=oss
# Checks: npm, pypi, cargo, github (no domains)
```

Available profiles:

| Profile     | Domains                                | Registries       | Handles |
| ----------- | -------------------------------------- | ---------------- | ------- |
| `startup`   | .com, .io, .dev, .app                  | npm, pypi        | github  |
| `developer` | .com, .io, .dev, .app, .sh, .org, .net | npm, pypi, cargo | github  |
| `oss`       | -                                      | npm, pypi, cargo | github  |
| `minimal`   | .com                                   | -                | -       |
| `website`   | .com, .org, .net                       | -                | -       |
| `web3`      | .xyz, .io, .gg                         | npm              | github  |

## Bug Fixes

### Expert Output (B3, B11)

The NAME column in `check --expert` output now consistently shows the checked
name. Previously displayed "ailink" due to a hardcoded literal in the output
formatter. Fixed in two passes: initial fix in `output/notes.go` (B3) and a
regression fix in `check.go` (B11).

### xAI Markup Stripping (B5)

Internal Grok citation markup (`<grok:render>`, `<argument>` tags) is now
stripped from response text before display. These tags were leaking through to
user-visible output in expert analysis results.

### Schema Validation (B12)

The `brand-plan` prompt no longer fails schema validation when called via
`generate` or `review`. The generate command was validating all responses
against the `search-response` schema, but `brand-plan` returns a different
response shape. Schema matching now uses the prompt-specific response schema.

### Check Defaults (B6)

Bare `namelens check <name>` without `--profile` now covers the full standard
suite: .com/.dev/.io/.app domains, npm/pypi/cargo registries, and GitHub
handle. Previously only checked .com.

### Standalone Binary Support

The CLI now works correctly when run from outside the repository directory.
Embedded config defaults and schema assets are used as fallback when
project-root discovery fails, so `make install` binaries work from any
directory.

### Deep Generation Quality

`--depth=deep` for `namelens generate` now uses broader naming strategies to
produce less correlated candidates. Quick depth continues to use focused
strategies for speed.

### Profile Slice Safety

Returned profile objects from `FindBuiltInProfile` no longer share backing
arrays with global built-in definitions. This prevents mutations to a returned
profile from corrupting the originals.

## Review Phonetics Passthrough

International phonetic analysis can now be scoped directly from the review
command:

```bash
namelens review myname --mode=core --locales en,de,ja --keyboards qwerty,qwertz
```

Previously required running separate `generate` commands with the
`name-phonetics` prompt.

## Upgrade Notes

No breaking changes. All v0.2.2 configurations remain valid.

**Behavior change**: Bare `namelens check <name>` now checks more targets than
before (4 TLDs + 3 registries + GitHub vs. just .com). Use `--profile=minimal`
for the previous behavior.

## Dogfooding

This release was validated through an end-to-end bolt-family naming exercise
(calendar and storage proxy naming for the Lanyte platform). The shakeout
verified all bug fixes across 9 expert checks, 3 brand reviews, and 2
context-file analyses with zero regressions.
