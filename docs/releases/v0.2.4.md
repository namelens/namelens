# NameLens v0.2.4 Release Notes

Release date: 2026-03-11

## Highlights

v0.2.4 is a stability hardening release: two bug fixes from standalone
deployment and expert bulk analysis, plus lint and security cleanup.

- **B13 fix**: Expert prompt loading now works outside git repos via embedded
  schema fallback
- **B14 fix**: Expert bulk rate-limit burst mitigated with serialized fallback,
  cooldown spacing, and exponential backoff
- **Code quality**: QF1012 lint cleanup and G115/G101 security hardening

## Expert Prompt Loading (B13)

**Status**: Fixed

Running `check --expert` from outside a git repository (e.g. from a
`make install` binary in `/usr/local/bin`) failed with "failed to load prompts"
because both `catalogForSchemas()` in `prompt/loader.go` and
`buildSchemaCatalog()` in `cmd/ailink_helpers.go` required a repository root to
locate schema files.

### Root Cause

The prompt loader and response schema catalog both used `findRepoRoot()` to
locate `schemas/ailink/v0/*.json` files. When running from an installed binary
outside the repo tree, `findRepoRoot()` returns an error and schema loading
fails entirely.

### Fix

Embedded prompt and response schemas via `go:embed` with temp-directory
extraction fallback, following the config layer's `standaloneAssetsRoot()`
pattern:

- `internal/ailink/prompt/loader.go`: Embeds `prompt.schema.json`, falls back to
  extracted temp dir when `findRepoRoot()` fails
- `internal/ailink/embedded_schemas.go`: New file embedding all
  `ailink/v0/*.json` response schemas via `embed.FS`, with `fs.WalkDir`
  extraction to temp dir
- `internal/cmd/ailink_helpers.go`: `buildSchemaCatalog()` falls back to
  `ailink.StandaloneSchemaCatalog()` when repo root unavailable

Makefile `sync-embedded-config` and `verify-embedded-config` targets extended to
keep embedded copies in sync with source-of-truth schemas under `schemas/`.

### Verification

```bash
# Install binary and run from outside repo
make install
cd /tmp
namelens check testname --expert
# Should succeed without "failed to load prompts" error
```

## Expert Bulk Rate-Limit Burst (B14)

**Status**: Fixed

The first name in a multi-name `--expert` batch occasionally got "provider
request failed" due to rate-limit burst immediately after the bulk request
completed.

### Root Cause

When `--expert-bulk` completes a batch request, individual fallback requests for
names not covered by bulk results fire immediately and concurrently. This burst
of requests can trigger provider rate limits (HTTP 429), causing the first
fallback name to fail.

### Fix

Layered mitigations in `internal/cmd/check.go`:

1. **Staggered scheduling**: Post-bulk fallback requests are scheduled with a 2s
   initial cooldown and 1.5s spacing from bulk completion time
2. **Serialized execution**: Fallback requests are mutex-serialized to prevent
   concurrent provider calls
3. **Exponential backoff**: Rate-limited (429) responses trigger retry with
   2s/4s/8s base backoff, up to 3 attempts, with deterministic per-name SHA256
   jitter (±300ms window)

### Verification

```bash
# Run multi-name expert check — all names should succeed
namelens check alpha beta gamma delta --expert --expert-bulk
```

## Code Quality

### QF1012 Lint Cleanup

Replaced `WriteString(fmt.Sprintf(...))` with `fmt.Fprintf()` across three
files:

- `internal/ailink/context/corpus.go` (13 instances)
- `internal/output/analysis.go` (4 instances)
- `internal/output/markdown.go` (4 instances)

### G115 Security Hardening

Added integer overflow bounds checks for unsafe integer conversions:

- `internal/daemon/discovery.go`: `uint64→int64` timestamp conversions,
  `int→uint16` port conversion
- `internal/daemon/lifecycle.go`: `int→uint32` PID conversions
- `internal/cmd/setup.go`: `#nosec G115` suppression on `uintptr→int` fd casts
  (stdlib `term` package pattern)

### G101 False Positive

Suppressed `G101` false positive on generated `ApiKeyScopes` constant in
`internal/api/openapi.gen.go` — the constant name contains "key" but holds no
credential material.

## Upgrade Notes

No breaking changes. All v0.2.3 configurations remain valid.
